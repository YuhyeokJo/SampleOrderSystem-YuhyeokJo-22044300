# PRD-step5.md - 출고 처리 기능 명세

## 개요
재고가 충분해져 승인 완료된(CONFIRMED) 주문을 확인하고, 담당자가 특정 주문을 선택해 출고를
실행한다. 출고가 처리되면 해당 주문은 완료 상태(RELEASE)로 전환된다. 콘솔(터미널) 기반으로
담당자가 직접 명령을 입력하여 사용한다.

## 기존 모델 확장

### `OrderModel` CONFIRMED 목록 조회 기능
- 저장된 주문 중 상태가 `CONFIRMED`인 주문만 조회하는 기능을 추가한다(예: `list_confirmed()`).
- CONFIRMED 상태의 주문이 없으면 빈 목록을 반환한다.

## 기능 명세

### 1. CONFIRMED 주문 목록 조회
- 입력: 없음
- 처리: `OrderModel.list_confirmed()`로 CONFIRMED 상태의 주문만 조회한다.
- 출력: 각 주문의 주문번호, 시료 ID, 고객명, 주문 수량, 상태(`CONFIRMED`)를 표 형태로 출력한다.
  CONFIRMED 상태의 주문이 없으면 목록이 비어 있음을 안내한다.

### 2. 출고 처리
- 입력: 주문번호
- 처리:
  1. 주문번호로 주문을 조회한다. 존재하지 않으면 오류를 안내하고 처리를 거부한다.
  2. 조회된 주문의 현재 상태가 `CONFIRMED`가 아니면(예: `RESERVED`, `PRODUCING`, `REJECTED`,
     이미 `RELEASE`된 주문) 오류를 안내하고 처리를 거부한다.
  3. 위 검증을 통과하면 주문 상태를 `RELEASE`로 전환한다(`OrderModel.update_status`).
  4. 출고 처리는 시료 재고나 생산라인에 어떠한 영향도 주지 않는다(재고 소비는 Step 4의 승인
     시점에 이미 처리되었으므로, 출고는 순수한 주문 상태 전환이다).
- 출력: 출고 완료 메시지(전환된 주문번호와 상태), 실패 시 사유 메시지

## 저장 방식
- 이번 step은 기존 Step 2/4의 주문 저장소를 그대로 재사용하며, 별도의 새로운 저장소를 추가하지
  않는다. 상태 전환은 기존 원자적 저장(`os.replace`) 방식을 통해 즉시 반영된다.

## 예외/경계 케이스 (테스트에 반드시 포함)
- 존재하지 않는 주문번호로 출고 시도 → 거부
- `RESERVED` 상태의 주문에 대해 출고 시도 → 거부
- `PRODUCING` 상태의 주문에 대해 출고 시도 → 거부
- `REJECTED` 상태의 주문에 대해 출고 시도 → 거부
- 이미 `RELEASE` 상태인 주문에 대해 다시 출고 시도 → 거부
- `CONFIRMED` 상태의 주문에 대해 출고 시도 → 성공, 상태가 `RELEASE`로 전환됨
- CONFIRMED 상태의 주문이 없는 상태에서 목록 조회 → 빈 결과 안내
- 여러 상태(RESERVED/CONFIRMED/PRODUCING/REJECTED/RELEASE)가 섞여 있을 때 목록 조회 결과에
  CONFIRMED 주문만 포함되는지 확인
- 출고 처리 후에도 대상 시료의 재고 수량이 변하지 않는지 확인
- 프로세스를 재시작한 뒤(새 인스턴스로) 출고로 변경된 주문 상태(`RELEASE`)가 그대로 남아 있는지
  확인

  정의하지 않는다.
