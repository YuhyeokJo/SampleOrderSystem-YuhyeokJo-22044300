# PRD-step4.md - 주문 승인/거절 기능 명세

## 개요
생산 라인 담당자가 접수된(RESERVED) 주문을 확인하고, 재고 상황에 따라 자동으로 승인 처리하거나
거절한다. 승인은 재고를 실제로 소비하는 행위이므로, 이번 step에서 시료 재고 차감 기능과 주문
목록 조회 기능을 추가로 구현한다. 콘솔(터미널) 기반으로 담당자가 직접 명령을 입력하여 사용한다.

## 기존 모델 확장

### `SampleModel` 재고 차감 기능
- 시료 ID와 차감할 수량을 받아 해당 시료의 재고 수량에서 차감하고 재저장한다
  (예: `decrease_stock(sample_id, amount)`).
- 차감할 수량은 항상 0 이상이며, 차감 후 재고가 음수가 되는 요청(현재 재고보다 큰 수량을
  차감하려는 시도)은 거부하고 오류를 안내한다.
- 존재하지 않는 시료 ID로 차감을 시도하면 오류를 안내하고 거부한다.

### `OrderModel` RESERVED 목록 조회 기능
- 저장된 주문 중 상태가 `RESERVED`인 주문만 조회하는 기능을 추가한다(예: `list_reserved()`).
- RESERVED 상태의 주문이 없으면 빈 목록을 반환한다.

## 승인/거절 처리 로직

### 공통 사전 검증
- 주문번호로 주문을 조회한다. 존재하지 않으면 오류를 안내하고 처리를 거부한다.
- 조회된 주문의 현재 상태가 `RESERVED`가 아니면(이미 승인/거절/생산/출고 처리된 주문이면)
  오류를 안내하고 처리를 거부한다. 승인과 거절 모두 이 규칙을 적용한다.

### 1. 주문 승인
- 입력: 주문번호
- 처리:
  1. 위 공통 사전 검증을 통과한다.
  2. 주문의 시료 ID로 `SampleModel`에서 현재 재고 수량을 조회한다.
  3. **재고가 주문 수량 이상**(재고 충분)이면:
     - `SampleModel.decrease_stock`으로 주문 수량만큼 재고를 차감한다.
     - 주문 상태를 `CONFIRMED`로 전환한다(`OrderModel.update_status`).
  4. **재고가 주문 수량 미만**(재고 부족)이면:
     - 부족분 = 주문 수량 − 현재 재고
     - 먼저 Step 3의 `ProductionModel`에 (주문번호, 시료 ID, 부족분)으로 생산 큐 등록을
       시도한다.
       - 등록이 실패하면(예: 시료의 수율이 정확히 0.0이어서 생산 자체가 불가능한 경우) 재고
         차감이나 주문 상태 변경을 전혀 적용하지 않고, 실패 사유를 안내하며 승인 처리를
         중단한다. 주문은 `RESERVED` 상태로 남는다.
       - 등록이 성공하면 `SampleModel.decrease_stock`으로 현재 남아 있는 재고 전량을 차감하여
         재고를 0으로 만들고, 주문 상태를 `PRODUCING`으로 전환한다(`OrderModel.update_status`).
       - 생산라인 등록으로 채워지는 부족분은 해당 주문 전용으로 소비되는 것으로 간주하며, 생산
         완료 시에도 시료의 일반 재고 수량에는 반영하지 않는다(Step 3의 생산 완료 처리 로직은
         수정하지 않는다).
- 출력: 처리 결과(CONFIRMED로 전환되었는지 PRODUCING으로 전환되었는지, 부족분이 있었다면
  생산라인에 등록된 수량)를 안내하는 메시지, 실패 시 사유 메시지

### 2. 주문 거절
- 입력: 주문번호
- 처리:
  1. 공통 사전 검증을 통과한다.
  2. 주문 상태를 즉시 `REJECTED`로 전환한다(`OrderModel.update_status`). 재고나 생산라인에는
     아무런 영향을 주지 않는다.
- 출력: 거절 완료 메시지, 실패 시 사유 메시지

### 3. 접수된 주문 목록 조회
- 입력: 없음
- 처리: `OrderModel.list_reserved()`로 RESERVED 상태의 주문만 조회한다.
- 출력: 각 주문의 주문번호, 시료 ID, 고객명, 주문 수량, 상태(`RESERVED`)를 표 형태로 출력한다.
  RESERVED 상태의 주문이 없으면 목록이 비어 있음을 안내한다.

## 저장 방식
- 이번 step은 기존 Step 1(시료)/Step 2(주문)/Step 3(생산 큐) 저장소를 그대로 재사용하며, 별도의
  새로운 저장소를 추가하지 않는다. 모든 변경(재고 차감, 주문 상태 전환, 생산 큐 등록)은 각
  저장소의 기존 원자적 저장(`os.replace`) 방식을 통해 즉시 반영된다.

## 예외/경계 케이스 (테스트에 반드시 포함)
- 존재하지 않는 주문번호로 승인/거절 시도 → 각각 거부
- 이미 `CONFIRMED`/`PRODUCING`/`REJECTED` 상태인 주문을 다시 승인/거절하려는 시도 → 거부
- 재고가 주문 수량과 정확히 같은 경계값 → 재고 충분으로 처리(즉시 CONFIRMED, 재고는 0으로 차감)
- 재고가 주문 수량보다 1 적은 경우 → 재고 부족으로 처리(부족분 1만큼 생산라인 등록, PRODUCING
  전환)
- 재고가 0인 시료에 대한 승인 → 부족분이 주문 수량 전체로 계산되어 생산라인에 등록되는지 확인
- 수율이 정확히 0.0인 시료에 대해 재고 부족으로 승인을 시도하는 경우 → 생산라인 등록이 거부되어
  승인 전체가 취소되고, 재고 차감이나 주문 상태 변경이 전혀 발생하지 않았는지 확인
- 승인 성공 후(재고 충분/부족 각 경우) 시료의 재고 수량이 정확히 차감된 값으로 갱신되었는지 확인
- 재고 부족으로 승인된 주문에 대해 생산이 완료된 뒤에도 시료의 일반 재고 수량이 변하지 않는지
  확인(Step 3 생산 완료 로직과의 연동 확인)
- RESERVED 상태의 주문이 없는 상태에서 목록 조회 → 빈 결과 안내
- 여러 상태(RESERVED/CONFIRMED/PRODUCING/REJECTED)가 섞여 있을 때 목록 조회 결과에 RESERVED
  주문만 포함되는지 확인
- 프로세스를 재시작한 뒤(새 인스턴스로) 승인/거절로 변경된 주문 상태와 차감된 재고 수량이 그대로
  남아 있는지 확인

## 범위 제외
- 생산 현황/대기 주문 조회, 생산 진행 기록은 Step 3에서 이미 구현되어 있으므로 이 문서에서
  다시 정의하지 않는다.
- 출고 처리, 모니터링, 메인 메뉴 통합은 이 문서의 범위가 아니다.
