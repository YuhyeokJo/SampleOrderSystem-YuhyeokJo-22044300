# Plan.md - 반도체 시료 생산주문관리 시스템 전체 개발 계획

전체 시스템을 몇 개의 step으로
나누어 구현할지에 대한 상위 계획을 정의한다. 각 step의 세부 구현 전에는 CLAUDE.md의 절차에 따라
`plans/Plan-step#.md`와 `prds/PRD-step#.md`를 각각 작성하고 검토를 받은 후 진행한다.

## Step 구성 및 근거

시료 등록 → 주문 접수 → 생산 → 승인/거절 → 출고 → 모니터링/메뉴 순으로 기능 간 의존관계가
존재하므로, 하위 도메인을 먼저 구현하고 이를 조합하는 순서로 step을 나눈다.

| Step | 이름 | 핵심 내용 |
|---|---|---|
| 1 | 시료 관리 | Sample 도메인 모델(시료 ID, 이름, 평균 생산시간, 수율, 재고 수량), 시료 등록, 목록 조회, 이름 검색 |
| 2 | 시료 주문 접수 | Order 도메인 모델, 주문번호 자동 생성(`ORD-YYYYMMDD-XXXX`), RESERVED 상태로 접수 |
| 3 | 생산라인 | 실 생산량 계산(`ceil(부족분 / (수율 × 0.9))`), 총 생산 시간 계산, FIFO 생산 큐, 생산 완료 처리(PRODUCING → CONFIRMED), 생산 현황/대기 주문 조회 |
| 4 | 주문 승인/거절 | RESERVED 목록 조회, 재고 충분 시 즉시 CONFIRMED 전환, 재고 부족 시 생산라인 등록 후 PRODUCING 전환, 거절 시 REJECTED 전환 |
| 5 | 출고 처리 | CONFIRMED 목록 조회, 출고 실행 시 RELEASE 전환 |
| 6 | 모니터링 | 상태별 주문 수 집계(RESERVED/CONFIRMED/PRODUCING/RELEASE, REJECTED 제외), 시료별 재고 상태(여유/부족/고갈) 표시 |
| 7 | 메인 메뉴 통합 | 콘솔 CLI 진입점, 메뉴 라우팅(시료 관리/시료 주문/주문 승인·거절/모니터링/생산라인/출고 처리), 전체 시료 요약 정보 표시 |
| 8 | 콘솔 UI 개선 | 구분선/표/상태 배지/진행률 바 등 공통 포맷 헬퍼 도입, 7개 view의 출력 포맷 개선(model/controller 로직 변경 없음) |

### Step 8 추가 경위
1~7단계 완료 후, 참고 자료(개인과제 안내 자료의 예시 UI 화면)를 바탕으로 콘솔 출력 형식을
개선해달라는 추가 요청에 따라 Step 8을 신설했다. text.md의 기능 요구사항이 아닌 순수 UI
포맷팅 개선이며, 세부 범위와 제외 항목은 `plans/Plan-step8.md`에 정의한다.

### Step 3을 Step 4보다 먼저 배치한 이유
주문 승인 시 재고가 부족하면 생산 라인에 자동 등록해야 하므로, 승인/거절 로직이 생산라인 기능을
호출하게 된다. 따라서 생산라인(Step 3)을 먼저 구현하여 승인/거절(Step 4)이 이를 참조할 수 있도록 한다.

### 모니터링·메인 메뉴를 마지막에 배치한 이유
모니터링(Step 6)과 메인 메뉴(Step 7)는 앞선 모든 기능의 데이터와 흐름을 조회·조립하는 성격이므로,
개별 도메인 기능이 모두 준비된 이후에 구현한다.

## 공통 원칙

- 각 step은 PRD-step#.md에 명시된 내용만을 구현 범위로 한다(제약 사항: "명시되지 않은
  기능은 구현하지 않음").
- 각 step 구현 전 `plans/Plan-step#.md` 작성 → 검토 → 승인 → (구현 후) 커밋.
- 각 step 세부 기능은 `prds/PRD-step#.md` 작성 → 검토 → 구현.
- 모든 구현은 CLAUDE.md에 정의된 agent 파이프라인
  (`consistency-verifier` → `ai-action` → `test-verifier` ∥ `compliance-verifier`)을 따른다.
- 이 전체 계획(Step 구성 및 순서)은 향후 세부 설계 과정에서 필요 시 조정될 수 있으며, 변경 시 이
  문서를 갱신한다.
